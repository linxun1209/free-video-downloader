# 用户个人信息编辑功能设计

## 背景

当前项目已经具备用户注册、登录、鉴权、VIP 状态展示等基础能力，但缺少用户个人信息查看和编辑能力。现有前端在顶部导航中提供用户下拉菜单，登录相关操作通过弹窗完成；后端以 FastAPI + SQLite 提供认证接口，用户对象目前仅包含邮箱和 VIP 相关字段。

本次需求是在不改变整体产品结构的前提下，补齐“用户个人信息编辑”能力，并保持与现有 UI 风格和交互模式一致。

## 目标

- 为已登录用户提供个人信息编辑入口
- 支持编辑 `nickname`、`phone`、`bio`
- `email` 仅展示，不允许编辑
- 前后端均完成实现，保存后立即同步最新用户信息
- 保持现有白底、圆角、轻阴影、弹窗式交互风格

## 非目标

- 不支持头像上传
- 不支持修改邮箱
- 不支持修改密码
- 不新增独立“个人中心”页面
- 不引入新的状态管理库或路由系统

## 方案选择

### 方案 A：顶部用户菜单新增“个人信息”，点击后打开编辑弹窗

这是本次采用方案。

优点：
- 与现有 `AuthModal` 的交互模式一致
- 不需要新增页面或路由
- 改动范围小，能快速上线
- 用户入口清晰，符合当前导航结构

缺点：
- 后续若扩展更多账号设置项，弹窗承载能力有限

### 方案 B：新增独立“个人中心”页面

优点：
- 更适合后续扩展账号设置、订单历史、密码修改等内容

缺点：
- 当前项目没有页面路由体系，新增成本偏高
- 与本次需求规模不匹配

### 方案 C：在用户下拉菜单内直接嵌入编辑表单

优点：
- 实现最省

缺点：
- 表单空间受限
- 可用性较差
- 与现有交互质量不一致

## 用户体验设计

### 入口

- 仅在已登录状态显示
- 位于顶部导航右侧用户头像下拉菜单中
- 菜单项文案为“个人信息”
- 展示位置在“退出登录”上方，和“开通 VIP”并列为账号相关操作

### 弹窗

- 交互形式复用现有认证弹窗模式：页面中央浮层 + 半透明遮罩
- 视觉风格与 `AuthModal.vue` 保持一致：
  - 白底卡片
  - 大圆角
  - 轻阴影
  - 统一输入框高度、边框、聚焦态
  - 顶部标题 + 描述文案

### 表单字段

- 邮箱地址：只读展示，不允许修改
- 昵称：单行输入，选填
- 手机号：单行输入，选填
- 个人简介：多行输入，选填

### 保存反馈

- 保存按钮在提交中显示 loading 状态
- 表单校验失败时，在弹窗内展示错误信息
- 保存成功后：
  - 关闭弹窗
  - 更新前端当前用户状态
  - 更新本地缓存用户信息
  - 用户下拉菜单中的展示信息同步刷新

## 数据模型设计

### users 表新增字段

- `nickname TEXT`
- `phone TEXT`
- `bio TEXT`

### 兼容性

- 由于项目使用 SQLite 且已有历史数据，初始化逻辑需要兼容已有库
- 若表已存在但新字段不存在，应通过迁移式补列方式安全添加
- 旧用户数据默认值为空，不影响既有登录与 VIP 功能

## 后端接口设计

### 统一用户响应结构

扩展当前用户响应对象，在保留原字段基础上新增：

- `nickname`
- `phone`
- `bio`

返回对象示意：

```json
{
  "id": 1,
  "email": "user@example.com",
  "nickname": "张三",
  "phone": "13800138000",
  "bio": "喜欢收藏高质量视频素材",
  "is_vip": false,
  "vip_expire_at": null
}
```

### 新增接口

#### `GET /api/auth/profile`

用途：
- 获取当前登录用户的完整个人信息

鉴权：
- 必须登录

返回：
- 与扩展后的用户响应结构一致

#### `PUT /api/auth/profile`

用途：
- 更新当前登录用户的个人信息

请求体：

```json
{
  "nickname": "张三",
  "phone": "13800138000",
  "bio": "喜欢收藏高质量视频素材"
}
```

字段规则：
- 三个字段均为可选字符串
- 空字符串允许提交，服务端会做 `trim`
- 超出长度限制时返回 400

建议校验：
- `nickname` 最大 30 字符
- `phone` 最大 20 字符
- `bio` 最大 200 字符

返回：
- 更新后的完整用户信息

### 后端实现拆分

- `database.py`
  - 为 `users` 表初始化补充新字段
  - 新增更新用户资料的方法
- `api_auth.py`
  - 增加 `GET /profile`
  - 增加 `PUT /profile`
  - 扩展 `_build_user_response`
- 现有 `GET /me`、登录、注册返回的用户对象一并包含新增字段，保证前端状态结构统一

## 前端实现设计

### API 层

在 `frontend/src/api/auth.js` 中新增：

- `fetchProfile()`
- `updateProfile(payload)`

并保持：
- 登录后保存用户对象到 localStorage
- 更新资料成功后覆盖本地缓存

### 组件设计

新增独立组件 `ProfileModal.vue`，职责如下：

- 展示个人信息表单
- 拉取/接收当前用户数据
- 管理表单输入、提交态、错误态
- 保存成功后向父组件抛出更新事件

组件输入输出建议：

- `visible`
- `user`
- `loading`
- 事件：
  - `close`
  - `success`

### 顶部导航集成

在 `AppHeader.vue` 中：

- 用户下拉菜单增加“个人信息”菜单项
- 点击后触发 `open-profile` 事件

在 `App.vue` 中：

- 新增个人信息弹窗开关状态
- 监听 `open-profile`
- 接收更新后的用户对象并写回 `currentUser`

## 数据流

### 初始加载

1. 用户登录后，前端保存 token 和用户对象
2. 页面刷新时继续通过 `fetchMe()` 恢复用户状态
3. 因 `me` 响应已扩展，恢复后的用户对象天然包含资料字段

### 编辑流程

1. 用户打开顶部菜单
2. 点击“个人信息”
3. 弹出资料编辑弹窗，并以当前用户数据作为初始值
4. 用户修改昵称、手机号、个人简介
5. 点击保存
6. 前端调用 `PUT /api/auth/profile`
7. 后端校验并写入数据库
8. 后端返回更新后的完整用户对象
9. 前端更新 `currentUser` 和 localStorage
10. 关闭弹窗并刷新头部展示

## 错误处理

### 后端

- 未登录：返回 401
- 字段类型非法：返回 400
- 长度超限：返回 400，并给出中文错误信息

### 前端

- 请求失败时在弹窗内展示错误提示
- 提交期间禁用按钮与输入框，避免重复提交
- 不影响现有登录态；即使保存失败，页面其余功能正常可用

## 展示策略

顶部菜单中的摘要展示保持克制：

- 继续以邮箱作为主识别信息，保证稳定性
- 个人信息编辑成功后，不要求在头像字母逻辑中强依赖昵称
- 头像字母仍可优先沿用邮箱首字母，避免昵称为空时产生额外分支复杂度

如后续要增强体验，可再考虑：
- 昵称优先展示
- 简介预览
- 资料完成度提示

这些均不在本次范围内。

## 测试设计

### 后端

应覆盖：
- 已登录用户获取资料成功
- 已登录用户更新资料成功
- `nickname` 超长时失败
- `phone` 超长时失败
- `bio` 超长时失败
- 未登录访问资料接口失败

### 前端

至少验证：
- 已登录用户下拉菜单出现“个人信息”入口
- 弹窗默认展示当前用户资料
- 保存成功后关闭弹窗并同步更新用户状态
- 接口报错时显示错误信息

如果仓库当前没有完善测试基础设施，则本次至少完成手工验证路径，并补充必要的最小自动化验证。

## 风险与约束

- SQLite 不支持直接在 `CREATE TABLE IF NOT EXISTS` 中自动补已有表字段，因此必须显式检查列是否存在
- 当前项目前端未引入复杂全局状态管理，用户信息同步必须继续以 `App.vue` 为单一状态源
- 现有用户对象结构被多个组件使用，新增字段时应保持向后兼容，不改变原有字段语义

## 实施边界

本次实现限定为：

- 数据库补字段
- 后端新增资料读取和更新接口
- 用户响应结构扩展
- 前端新增个人信息编辑弹窗
- 顶部菜单接入入口
- 本地用户状态同步

不包含任何与支付、下载、AI 总结逻辑相关的改动。
